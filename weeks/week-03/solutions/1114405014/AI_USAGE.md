# AI 使用說明

在本專案的開發過程中，我使用生成式 AI（ChatGPT）作為輔助工具，以協助理解題目需求、設計程式架構、檢查測試案例與整理文件內容。

AI 並未直接完成整個專案，而是作為開發過程中的輔助工具，主要用於討論設計策略、分析測試案例以及整理文件格式。

核心程式邏輯仍由我自行實作並透過測試驗證。

---

# AI 使用階段

本專案 AI 的使用主要集中在以下幾個階段：

## 1. 專案架構討論

在開始實作前，透過 AI 討論專案應如何拆分模組。

最終採用以下架構：

```
robot_core.py → 核心邏輯
robot_game.py → pygame 視覺化
tests/ → 單元測試
```

此架構的優點：

- 核心邏輯與 UI 分離
- 核心模組可單獨測試
- pygame 僅負責視覺化與操作

這樣可以讓核心邏輯更容易維護與測試。

---

## 2. TDD 開發流程討論

AI 協助規劃 TDD 的開發流程：

```
撰寫測試
↓
執行測試 → Red
↓
實作最小功能
↓
再次測試 → Partial Green
↓
修正測試案例
↓
全部測試通過 → Green
```

這個流程幫助我先設計測試案例，再逐步實作功能。

---

## 3. 測試案例問題分析

在測試過程中，發現一個測試案例：

`test_same_position_same_direction_with_scent_should_continue_to_next_command`

原本測試使用：

```
FRF
```

AI 協助分析後發現：

初始位置: (5,3,N)  
scent: {(5,3,N)}

F → 因 scent 忽略  
R → 方向變為 E  
F → (6,3) 超出邊界 → LOST

因此測試失敗並不是程式錯誤，而是測試案例設計不正確。

後來將測試修改為：

```
LFF
```

執行流程：

(5,3,N)

L → (5,3,W)  
F → (4,3,W)
F → (3,3,W)


機器人可正常移動且不會 LOST，成功驗證 scent 機制。

# 採用的 AI 建議

本專案採用了 AI 的以下建議：

### 1. 使用 dataclass

```python
@dataclass
class Robot:
    x: int
    y: int
    direction: str
    lost: bool = False

優點：

程式碼更簡潔

狀態管理清晰

更容易撰寫測試

2. scent 使用 set 資料結構

AI 建議使用：

set[(x, y, direction)]

原因：

查詢時間複雜度為 O(1)

能同時記錄位置與方向

可避免重複建立 scent

3. 核心邏輯與 UI 分離

AI 建議將：

robot_core.py
robot_game.py

分離。

這樣可以：

單獨測試核心邏輯

pygame 不影響測試

提高程式結構清晰度

拒絕的 AI 建議

在開發過程中，我也沒有完全採用所有 AI 建議。

其中一個例子是：

AI 曾建議先實作 robot_game.py 的 pygame 介面，再逐步補齊核心邏輯。

但我選擇先完成 robot_core.py 與所有單元測試，原因是：

核心邏輯應先被測試驗證

UI 不應影響核心測試

符合 TDD 開發原則

因此我先完成：

tests
↓
robot_core.py
↓
測試全部通過
↓
最後才實作 robot_game.py
AI 建議不完整並自行修正

AI 在分析測試案例時提供了初步方向，但實際測試案例的修正仍需要自行驗證。

例如：

原測試：

FRF

AI 提出可能問題，但最終測試修正為：

LFF

並透過實際執行測試確認邏輯正確。

這表示 AI 提供了分析方向，但仍需自行驗證與修正。
